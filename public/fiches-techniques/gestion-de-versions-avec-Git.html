Gestion de versions avec Git

Définition concise

Git est un système de gestion de versions qui permet de suivre les modifications du code, collaborer avec d’autres développeurs et revenir à des états antérieurs du projet si nécessaire. Il est essentiel pour organiser le développement de manière sécurisée et structurée.

Contexte d'utilisation

  • Suivre les modifications du code et pouvoir revenir en arrière
  • Collaborer efficacement avec plusieurs développeurs
  • Organiser le développement en branches pour tester de nouvelles fonctionnalités
  • Documenter l’évolution du projet à travers des messages de commit clairs

Cas d'usage réel

Dans mon projet Portfolio2025, j’ai appliqué Git ainsi :

  • Création d’un dépôt Git local et synchronisation avec GitHub
  • Organisation du développement via des branches : main pour la version stable et feature/nom-feature pour les nouvelles fonctionnalités
  • Commits fréquents avec des messages descriptifs (« Ajout de la fiche workflow », « Correction responsive header »)
  • Merge des branches testées vers main après validation

Pièges à éviter

  • Commits trop volumineux ou peu explicites
  • Travailler directement sur la branche principale (main) sans tests
  • Oublier de synchroniser avec le dépôt distant, ce qui crée des conflits
  • Ne pas nettoyer les branches après fusion, ce qui encombre le projet

Analyse personnelle

Au début, je faisais de grands commits globaux qui rendaient le suivi des changements difficile. Après avoir adopté des commits plus petits et descriptifs, la traçabilité s’est nettement améliorée.

L’utilisation de branches pour chaque fonctionnalité m’a permis de tester et corriger le code sans impacter la version stable. Cette méthode m’a appris l’importance de la rigueur et de l’organisation dans un projet même de petite taille.

Git m’a aussi montré l’intérêt de documenter mon travail : chaque commit devient un résumé de l’évolution du projet.
